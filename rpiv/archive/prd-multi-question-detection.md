# 多题图片智能识别与选题

## 1. 执行摘要

AIXue（爱学）当前的解题流程假设每张图片只包含一道题目。但实际使用中，学生拍照上传的试卷或作业往往包含多道题目，导致 OCR 识别混乱、解答质量下降、不完整题目浪费 LLM 调用资源。

本功能在现有解题流程中增加"多题检测"环节：上传图片后先用 LLM 分析图中题目数量、编号和完整性，再根据结果走不同路径——单题直接解答、多题让用户按钮选择、全部不完整提示重拍。

**MVP 目标**：让用户上传含多道题目的图片时，能够智能识别并按需选择解答，消除当前"多题混答"的体验问题。

## 2. 使命

**使命声明**：让每一次拍照上传都能精准定位到学生真正想解的那道题。

**核心原则**：
1. **零摩擦单题体验** — 单题图片不增加任何额外步骤，用户无感知
2. **智能过滤** — 自动识别不完整题目，避免无效解答浪费时间和成本
3. **用户掌控** — 多题场景下由用户决定解哪道题，而非系统猜测
4. **渐进增强** — 在现有解题流程上增量叠加，不破坏已有功能

## 3. 目标用户

- **主要用户**：初中/高中学生
- **使用场景**：手机拍照上传试卷、作业、练习册中的题目
- **技术舒适度**：熟悉移动端操作，习惯对话式交互
- **关键痛点**：
  - 拍照时难以精确裁剪到单题，经常拍到多道题
  - 多题混合解答质量差，答非所问
  - 不完整题目浪费等待时间

## 4. MVP 范围

### 范围内

- ✅ 多题检测：通过 LLM 分析图片中的题目数量、编号、完整性
- ✅ 单题快速通道：检测到 1 道完整题时自动进入解题，无额外交互
- ✅ 多题选择 UI：对话中显示按钮组，支持选择单题或"全部解答"
- ✅ 不完整题目标记：灰显 + "不完整"标签，不可选择
- ✅ 全部不完整提示：提示用户重新拍照
- ✅ user_hint 兼容：有文字说明时跳过多题检测，走现有聚焦识别流程
- ✅ 单次最多 5 题限制，超出提示分批拍照
- ✅ 逐题解答：选择"全部解答"时按顺序逐题调用解题流程，每题有分隔标记

### 范围外

- ❌ 图片裁剪/校正（后续单独做）
- ❌ 用户手动框选题目区域
- ❌ 选题超时自动处理
- ❌ 不完整题目内容展示
- ❌ 多张图片合并处理

## 5. 用户故事

**US-1**：作为学生，我上传一张包含多道题目的试卷照片，我想要选择其中某道题来解答，以便不用重新裁剪拍照。

**US-2**：作为学生，我上传一张只有一道题的图片，我想要系统自动开始解答，以便和以前一样快速获得帮助。

**US-3**：作为学生，我上传的图片中有些题被拍得不完整，我想要系统告诉我哪些题不完整，以便我知道需要重新拍这些题。

**US-4**：作为学生，我上传图片并附带文字"第14题"，我想要系统直接解答第14题，以便跳过选题步骤。

**US-5**：作为学生，我想一次解答图片中的所有题目，以便快速复习整张试卷。

**US-6**：作为学生，我上传的图片中所有题目都不完整，我想要收到明确的提示，以便知道需要重新拍一张更清晰的照片。

**US-7**：作为学生，图片中题目超过 5 道时，我想要得到分批拍照的建议，以便每次都能获得高质量的解答。

## 6. 核心架构与模式

### 现有解题流程

```
用户上传 → solver.solve() → _recognize(OCR) → detect_subject() → MathSolver/GeneralSolver → 保存记录
```

### 新增多题检测后的流程

```
用户上传图片（无 user_hint）
  → 多题检测 LLM 调用（新增）
  → 结果分支:
    - 1 道完整题 → 现有 OCR + 解题流程（无感知）
    - 2-5 道完整题 → 返回题目列表 + 按钮组（新增 API 响应类型）
    - 0 道完整题 → 返回提示信息
    - 超过 5 题 → 返回前 5 题 + 分批拍照提示

用户选择题目
  → 单题：聚焦 OCR（复用 RECOGNITION_FOCUSED_PROMPT）→ 解题
  → 全部：逐题循环调用解题流程 → 合并结果

用户上传图片（有 user_hint）
  → 跳过多题检测 → 现有聚焦 OCR 流程（不变）
```

### 关键设计决策

1. **多题检测作为独立 LLM 调用**：在 OCR 之前新增一次 LLM 调用，使用 light 模型（成本低），输出结构化 JSON
2. **复用聚焦 OCR**：用户选题后，通过已有的 `RECOGNITION_FOCUSED_PROMPT` + `user_hint` 机制聚焦识别
3. **按钮组作为特殊消息类型**：前端新增消息类型 `question_select`，包含题目列表和回调按钮

## 7. 功能规格

### 7.1 多题检测服务

- 输入：图片数据 + media_type
- 输出：结构化 JSON，包含每道题的编号、摘要（前 20 字）、完整性标记
- 使用 `llm_model_light`（低成本快速模型）
- LLM 输出格式示例：

```json
{
  "questions": [
    {"number": "1", "summary": "已知函数 f(x) = x^2 + 2x...", "complete": true},
    {"number": "2", "summary": "如图所示，在三角形ABC中...", "complete": true},
    {"number": "3", "summary": "求证：当 a > 0 时...", "complete": false}
  ]
}
```

### 7.2 选题 API

- 新增 `/solve/select` 端点或扩展现有 `/solve` 端点返回类型
- 多题检测结果临时存储（内存/会话级别），用户选题时无需重复检测
- 选择"全部解答"时服务端逐题调用解题器

### 7.3 前端选题 UI

- 消息气泡中嵌入按钮组
- 每个按钮显示题号 + 摘要预览
- 不完整题目灰显，不可点击，显示"不完整"标签
- "全部解答"按钮放在列表底部
- 按钮点击后禁用（防重复提交），替换为"正在解答..."状态

### 7.4 逐题解答（全部解答模式）

- 按题号顺序逐题调用解题流程
- 每题解答结果以 "**第X题**" 标题分隔
- 某题解答失败时跳过继续，最后汇总提示哪些题失败
- 所有题目共用同一个 SolvingSession

## 8. 技术栈

沿用现有技术栈，无需引入新依赖：

- **后端**：FastAPI + SQLAlchemy (async) + Pydantic
- **前端**：Next.js 16 + React 19 + TypeScript + Tailwind CSS 4
- **LLM**：OpenRouter（`llm_model_light` 用于多题检测）
- **数据库**：PostgreSQL（生产）/ SQLite（测试）

## 9. 安全与配置

- **认证**：复用现有 JWT 认证，所有新端点需 `get_current_user` 依赖
- **配置**：无需新增环境变量，多题检测使用现有 `llm_model_light`
- **安全边界**：
  - 多题检测 prompt 不包含用户可控内容（仅图片）
  - 题目数量限制为 5，防止恶意上传触发大量 LLM 调用
  - 按钮选择通过题号索引传递，非用户自由文本输入

## 10. API 规格

### 现有 `/solve` 端点行为变更

**请求**：不变（image, text, subject, mode, session_id）

**响应扩展**：当检测到多题时，返回新的响应类型：

```json
{
  "type": "question_select",
  "questions": [
    {"number": "1", "summary": "已知函数 f(x)...", "complete": true},
    {"number": "2", "summary": "如图所示...", "complete": true},
    {"number": "3", "summary": "求证...", "complete": false}
  ],
  "message": "检测到 3 道题目，请选择要解答的题目",
  "session_id": "temp-session-id"
}
```

### 新增 `/solve/select` 端点

**请求**：
```json
{
  "session_id": "temp-session-id",
  "selected": [1]          // 题号索引数组，或 "all" 表示全部
}
```

**响应**：复用 `SolveResponse`，全部解答时 content 包含分隔标记的多题结果。

## 11. 成功标准

### MVP 成功定义
- 用户上传多题图片后能准确识别题目数量和完整性
- 单题图片不引入额外延迟或交互步骤
- 选题 UI 直观易用，点击即解

### 功能要求
- ✅ 单题图片与现有流程体验一致
- ✅ 多题图片正确返回选题按钮组
- ✅ 不完整题目被正确标记且不可选择
- ✅ user_hint 场景跳过多题检测
- ✅ 全部解答按顺序完成且结果清晰分隔
- ✅ 超过 5 题时给出分批建议

### 质量指标
- 多题检测准确率 > 90%（题目数量和完整性判断）
- 多题检测延迟 < 3 秒（使用 light 模型）
- 无回归：现有单题解题流程不受影响

## 13. 未来考虑

- **图片裁剪**：检测到多题后自动裁剪出每道题的区域，提供更精准的 OCR
- **多图合并**：支持上传多张图片组成完整试卷
- **题目缓存**：相同图片的检测结果缓存，减少重复 LLM 调用
- **批量导入题库**：多题检测结果直接导入个人题库

## 14. 风险与缓解措施

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 多题检测增加延迟和成本 | 每次上传多一次 LLM 调用 | 使用 light 模型（成本低、速度快）；单题场景下检测本身也很快 |
| LLM 返回非结构化结果 | 解析失败导致流程中断 | JSON 输出格式校验 + fallback：解析失败时回退到现有单题流程 |
| 多题检测准确率不足 | 误判题目数量或完整性 | Prompt 迭代优化；提供"重新检测"按钮；用户反馈收集 |
| 全部解答耗时过长 | 5 道题逐题解答可能超过 30 秒 | 前端展示逐题进度；考虑并行解答（后续优化） |
| 检测结果临时存储的生命周期管理 | 内存泄漏或结果过期 | 设置 TTL（如 10 分钟）；使用会话级存储而非全局存储 |
