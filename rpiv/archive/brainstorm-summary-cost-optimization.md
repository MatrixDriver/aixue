## 需求摘要

### 产品愿景
- **核心问题**：当前解题流程中，高分辨率图片直接送视觉 LLM 导致 token 消耗过高，且 OCR 和推理共用同一模型无法独立优化成本
- **价值主张**：通过前端图片预处理 + 后端 OCR/推理拆分 + 独立模型配置，在不牺牲识别质量的前提下大幅降低单题 API 成本
- **目标用户**：使用 AIXue 的初高中学生（终端用户无感知）+ 运营人员（通过 .env 配置模型）
- **产品形态**：前端 Next.js 组件 + 后端 FastAPI 服务改造，用户无感知的内部优化

### 核心场景（按优先级排序）
1. **图片自动压缩**：用户上传高清拍照/截图后，前端自动压缩至 1536px + WebP 格式再发送
2. **OCR/推理两阶段拆分**：后端将图片识别和解题推理拆为独立步骤，各自使用独立配置的模型
3. **模型独立配置**：运营通过 .env 分别配置 OCR 模型和推理模型

### 产品边界
- **MVP 范围内**：
  - 前端自动压缩（分辨率限制最长边 1536px + WebP quality=85）
  - 后端 OCR/推理拆分，中间通过结构化文本传递
  - .env 支持分别配置 OCR 模型（`LLM_MODEL_OCR`）和推理模型（`LLM_MODEL`）
  - 纯文本输入自动跳过 OCR 阶段
- **明确不做**：
  - 手动裁剪功能
  - 传统 OCR（PaddleOCR/Tesseract）方案
  - 按学科/难度分模型
  - 管理后台（已记录为独立 todo: feature-admin-dashboard.md）
  - 成本数据持久化到数据库（当前继续用日志记录 token 消耗）
- **后续版本考虑**：
  - 管理后台动态切换模型
  - OCR 识别质量自动检测 + 降级/升级策略
  - 成本统计面板
  - 按学科/难度智能选择推理模型

### 已知约束
- **质量优先**：OCR 阶段仍使用视觉 LLM（如 Gemini Flash），保证公式和图形识别准确率
- 前端预处理需兼容手机拍照和截图两种来源
- 图片格式转换在前端 Canvas API 完成，不增加后端负担
- 需通过 OpenRouter 调用模型（当前架构）

### 各场景功能要点

#### 场景1：图片自动压缩
- **功能点**：上传图片后，前端用 Canvas API 自动缩放至最长边 1536px，导出 WebP (quality=85)
- **关键交互**：完全透明，用户无感知。上传按钮行为不变，压缩在发送前自动完成
- **异常处理**：浏览器不支持 WebP 导出时降级为 JPEG (quality=80)；图片已小于 1536px 时不放大，仅做格式转换

#### 场景2：OCR/推理两阶段拆分
- **功能点**：
  - 阶段一（OCR）：接收图片，使用视觉 LLM（由 `LLM_MODEL_OCR` 配置）输出题目结构化文本（含 LaTeX 公式）
  - 阶段二（推理）：接收纯文本，使用推理 LLM（由 `LLM_MODEL` 配置）输出解题思路和答案
  - 纯文本输入直接进入阶段二
- **关键交互**：对用户透明，响应格式不变
- **异常处理**：OCR 阶段识别失败时返回明确错误提示；OCR 结果为空时提示用户重新上传清晰图片

#### 场景3：模型独立配置
- **功能点**：.env 新增 `LLM_MODEL_OCR` 配置项，与现有 `LLM_MODEL`（推理）独立；config.py 的 Settings 类新增对应字段
- **关键交互**：修改 .env 后重启服务生效
- **异常处理**：`LLM_MODEL_OCR` 未配置时默认回退到 `LLM_MODEL_LIGHT`（当前已有的轻量模型配置）
