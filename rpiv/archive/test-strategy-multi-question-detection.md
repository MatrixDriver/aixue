# 测试策略: 多题图片智能识别与选题

## 1. 测试范围

### 1.1 被测模块

| 模块 | 文件 | 测试重点 |
|------|------|----------|
| 多题检测服务 | `services/ocr_service.py`（新增方法） | LLM 返回的多题检测结果解析、完整性判断、数量限制 |
| 解题主控 | `services/solver_service.py` | 新增多题分支逻辑、user_hint 快捷路径、逐题解答编排 |
| API 端点 | `api/endpoints/solver.py` | 请求/响应格式变化、多题选择交互、按钮组数据结构 |
| 前端交互 | `frontend/src/` | 按钮组渲染、选题交互、逐题结果展示（E2E 范畴） |

### 1.2 不在测试范围

- 图片裁剪/校正功能（明确不做）
- 用户手动框选题目区域（明确不做）
- 选题超时自动处理（明确不做）
- LLM 模型本身的识别准确率（属于模型评估，非功能测试）

## 2. 测试层次

### 2.1 单元测试（重点，自动化）

**目标**: 验证多题检测的核心逻辑，与 LLM 调用解耦。

- **多题检测结果解析**: mock LLM 返回的 JSON/结构化文本 → 验证解析为题目列表
- **完整性判断逻辑**: 给定解析后的题目元数据 → 验证完整/不完整标记
- **数量限制**: 验证超过 5 题时只取前 5 道 + 生成提示信息
- **user_hint 快捷路径**: 有 hint 时跳过多题检测，沿用现有 RECOGNITION_FOCUSED_PROMPT
- **单题快速通道**: 检测到仅 1 道完整题时自动进入解题，无选题步骤
- **全部不完整场景**: 无完整题目时返回正确的错误提示
- **响应数据结构**: 验证 API 返回的按钮组数据符合约定格式

### 2.2 集成测试（中等优先级，自动化）

**目标**: 验证 SolverService 内部模块协作。

- **完整解题流程（单题）**: mock LLM → OCR 识别 → 多题检测（单题）→ 学科判定 → 解题 → 保存记录
- **完整解题流程（多题选择）**: mock LLM → 多题检测 → 返回选题按钮 → 用户选择 → 解题
- **逐题解答编排**: mock LLM → "全部解答" → 验证逐题调用 + 失败跳过 + 结果汇总
- **数据库持久化**: 多题场景下 SolvingSession 和 Message 的正确保存

### 2.3 API 测试（中等优先级，自动化）

**目标**: 验证 HTTP 接口的请求/响应契约。

- **多题检测 API**: 上传图片 → 返回题目列表 + 按钮组数据
- **选题提交 API**: 提交选择的题号 → 触发解题
- **全部解答 API**: 提交全部解答请求 → 逐题结果
- **错误场景**: 无效选择、空图片、超大图片等边界情况
- **user_hint 兼容**: 带 hint 的请求走快捷路径

### 2.4 E2E 测试（低优先级，手动）

**目标**: 验证前后端联动的用户体验。

- 上传多题图片 → 按钮组显示 → 点击选题 → 解答展示
- 不完整题目灰显 + 标签
- 全部解答的逐题展示 + 分隔标记

## 3. 测试策略

### 3.1 Mock 策略

多题检测功能高度依赖 LLM 调用，测试中必须 mock `LLMService` 的以下方法：

| 方法 | Mock 策略 |
|------|-----------|
| `recognize_image()` | 返回预设的 OCR 文本 |
| `chat()` | 根据 prompt 内容返回预设的多题检测 JSON / 学科判定结果 / 解题内容 |

**Mock 原则**:
- 使用 `unittest.mock.AsyncMock` 模拟异步方法
- 为每个场景准备独立的 mock 返回值 fixtures
- 不 mock 数据库层（使用 SQLite 内存数据库，与现有 conftest 一致）

### 3.2 测试数据

需要准备以下 fixtures：

1. **多题检测 LLM 返回模板**:
   - 单题完整
   - 多题（2-5 道）全部完整
   - 多题部分不完整
   - 全部不完整
   - 超过 5 题
   - 格式异常/解析失败

2. **图片 fixtures**: 复用现有 `sample_math_image` fixture（1x1 PNG），测试逻辑不依赖真实图片内容

### 3.3 测试文件组织

```
tests/
  test_services/
    test_multi_question_detection.py    # 多题检测解析 + 完整性判断单元测试
    test_solver_multi_question.py       # SolverService 多题流程集成测试
  test_api/
    test_solver_multi_question.py       # API 端点测试
```

## 4. 验收标准

### 4.1 功能验收

| 场景 | 验收条件 |
|------|----------|
| 单题快速通道 | 检测到 1 道完整题 → 无选题步骤，直接解答 |
| 多题选择 | 检测到 2-5 道题 → 返回按钮组数据，含题号 + "全部解答" |
| 不完整题过滤 | 不完整题标记 `incomplete: true`，不可被选择解答 |
| 全部不完整 | 返回提示信息，无按钮组 |
| 超过 5 题 | 只展示前 5 道 + 提示信息 |
| user_hint 快捷路径 | 有 hint 时跳过多题检测，走现有聚焦识别 |
| 全部解答 | 逐题解答，每题有分隔标记，失败跳过不中断 |

### 4.2 非功能验收

- 所有新增代码有对应测试覆盖
- 现有测试不因新功能引入而失败（回归）
- Mock 不泄漏到其他测试（fixture 隔离）

## 5. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| LLM 返回格式不稳定 | 多题检测解析失败 | 单元测试覆盖多种异常格式；服务层做容错降级 |
| 多题检测增加一次 LLM 调用 | 延迟增加 | 性能不在 MVP 测试范围，但代码审查关注不必要的重复调用 |
| 逐题解答中某题失败 | 中断整体流程 | 测试验证失败跳过逻辑 + 最终汇总 |
| 前端按钮组数据结构变更 | 前后端不一致 | API 测试固定响应 schema |

## 6. 测试执行计划

1. **阶段 1**: 等 PRD 完成 → 编写测试规格（T5）
2. **阶段 2**: 等实施计划完成 → 编写测试代码（T7）
3. **阶段 3**: 等代码实现完成 → 运行测试 + 代码审查（T8 + T9）
