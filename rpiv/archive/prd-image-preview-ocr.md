# 产品需求文档：图片上传预览 + OCR 结果展示优化

## 1. 执行摘要

AIXue 解题页面当前存在体验断裂问题：用户上传题目图片后，聊天气泡中仅显示"[图片]"文字，无法看到自己上传了什么内容，也无法查看 OCR 中间识别结果。用户对"AI 是否正确理解了题目"缺乏信心，导致信任感不足。

本需求通过三个核心改进解决该问题：(1) 用缩略图替代"[图片]"文字，让用户直观看到上传内容；(2) 提供展开面板展示 OCR 识别文本，让识别过程透明；(3) 支持点击放大查看原图细节。同时优化加载状态反馈，确保上传到解题全流程体验流畅。

**MVP 目标**：纯前端 UI 优化，不涉及后端接口变更，利用现有 `SolveResponse.question` 字段获取 OCR 文本，结合前端本地图片预览，实现完整的图片+OCR 展示体验。

## 2. 使命

**使命声明**：让图片上传和 OCR 识别过程对用户完全透明可见，建立用户对 AI 解题能力的信任。

**核心原则**：
1. **即时反馈**：图片上传后零延迟显示缩略图预览，不依赖后端响应
2. **信息透明**：OCR 识别文本可查看，用户能判断 AI 是否正确理解了题目
3. **渐进展示**：默认精简展示（缩略图），按需展开详情（OCR 文本），不增加视觉噪音
4. **优雅降级**：图片丢失（页面刷新）显示占位图，OCR 失败显示友好提示

## 3. 目标用户

- **主要角色**：初高中学生（13-18 岁）
- **技术舒适度**：熟悉手机拍照和即时通讯 App 交互，对"聊天气泡中显示图片"有天然预期
- **核心需求**：上传题目后能确认"AI 看到了什么"，建立对解题结果的信任
- **关键痛点**：
  - 上传图片后只看到"[图片]"，不确定图片是否上传成功
  - 无法知道 OCR 是否正确识别了题目文字
  - 想回看上传的题目时无处可看

### 范围内

**交互细节：**

### 范围外

- ❌ 后端 API 接口变更（现有 `SolveResponse.question` 已包含 OCR 文本）
- ❌ 图片持久化存储（R2 方案，单独功能）
- ❌ OCR 文本可编辑/修正
- ❌ 后端图片增强处理（旋转校正、对比度优化）
- ❌ 历史会话的图片回显（依赖图片持久化，不在本期范围）

## 5. 用户故事

1. **作为学生**，我想在聊天气泡中看到上传的题目图片缩略图，以便确认图片上传成功且内容正确。
   - *示例*：拍照上传一道数学题，气泡中立即显示题目照片的缩略图，而非"[图片]"文字。

2. **作为学生**，我想查看 OCR 识别出的题目文字，以便确认 AI 正确理解了题目内容。
   - *示例*：点击缩略图下方的"查看识别结果"按钮，展开面板显示 OCR 文本"已知函数 f(x) = x^2 + 2x + 1，求 f(x) 的最小值"。

3. **作为学生**，我想点击缩略图查看完整大图，以便确认图片清晰度和完整性。
   - *示例*：点击缩略图弹出全屏模态框，显示原始分辨率图片，点击空白处或按 ESC 关闭。

4. **作为学生**，我想在图片上传期间看到即时反馈，以便知道上传正在进行。
   - *示例*：选择图片后气泡立即显示缩略图，OCR 文本区域显示"识别中..."加载动画，AI 回复后加载动画替换为识别文本。

5. **作为学生**，我想在 OCR 识别失败时看到清晰的错误提示，以便知道需要重新上传更清晰的图片。
   - *示例*：图片模糊导致 OCR 失败，展开区域显示"未能识别图片中的文字，建议重新拍摄更清晰的照片"。

## 6. 核心架构与模式

### 架构方法

纯前端改造，不涉及后端变更。核心思路：

1. **扩展 Message 类型**：在前端 `Message` 接口新增 `imageUrl`（本地 blob URL）和 `ocrText` 字段
2. **改造 use-chat Hook**：发送图片消息时通过 `URL.createObjectURL()` 生成本地预览 URL 存入消息；收到 `SolveResponse` 后将 `question` 字段作为 `ocrText` 回填到用户消息
3. **新建组件**：`ImageMessageBubble`（缩略图+展开面板）、`ImageLightbox`（模态框大图查看）
4. **改造 ChatMessage**：识别含图片的用户消息，使用 `ImageMessageBubble` 渲染

### 数据流

```
用户选择图片
  → compressImage() 压缩
  → URL.createObjectURL() 生成本地预览 URL
  → 创建 userMsg（content="[图片]", imageUrl=blobURL）
  → 调用 solveQuestion API
  → 收到 SolveResponse
  → 从 response.question 提取 OCR 文本
  → 更新 userMsg.ocrText = response.question
```

### 关键设计决策

- **图片数据来源**：使用 `URL.createObjectURL()` 而非 `FileReader.readAsDataURL()`，避免大量 base64 数据占用内存
- **OCR 文本来源**：复用后端 `SolveResponse.question` 字段，无需新增 API
- **页面刷新后处理**：blob URL 失效，显示占位图（灰色图片图标 + "图片已过期"文字），OCR 文本通过状态保留可继续显示
- **展开面板默认收起**：减少视觉干扰，用户有需要时主动展开

## 7. 功能规格

### 7.1 缩略图展示

| 属性 | 规格 |
|------|------|
| 尺寸 | 宽度 200-240px，高度按比例自适应，最大高度 300px |
| 圆角 | 8px (rounded-lg) |
| 边框 | 1px solid gray-200 |
| 对象适配 | `object-cover`，保持图片比例 |
| 点击行为 | 打开 Lightbox 模态框 |
| 失败状态 | 灰色占位图 + "图片加载失败" |

### 7.2 展开面板（OCR 结果）

| 属性 | 规格 |
|------|------|
| 触发按钮 | 缩略图下方，文字"查看识别结果 ▼"/"收起 ▲" |
| 默认状态 | 收起 |
| 动画 | CSS transition，高度过渡 200-300ms |
| 布局 | 上方：较大尺寸图片（宽度 100%，最大 400px）；下方：OCR 文本 |
| OCR 文本样式 | 白色背景，灰色文字，等宽或 sans-serif 字体，带左边框指示 |
| 空/失败状态 | 显示"未能识别图片中的文字" |
| 加载状态 | 脉冲动画占位符 |

### 7.3 Lightbox 模态框

| 属性 | 规格 |
|------|------|
| 背景遮罩 | 半透明黑色 (bg-black/60) |
| 图片尺寸 | 最大宽度 90vw，最大高度 90vh，`object-contain` |
| 关闭方式 | 点击遮罩层 / 右上角关闭按钮 / ESC 键 |
| 动画 | 淡入淡出 |
| 无障碍 | `role="dialog"`, `aria-modal="true"`, focus trap |

### 7.4 加载状态

| 阶段 | 展示 |
|------|------|
| 图片选择后 | 缩略图立即显示（本地 File → blob URL） |
| API 请求中 | 展开面板的 OCR 文本区域显示骨架屏/脉冲动画 |
| API 响应后 | 加载动画替换为 OCR 文本内容 |
| 请求失败 | 加载动画替换为错误提示文字 |

## 9. 安全与配置

### 安全范围

**范围内：**
- blob URL 仅在当前页面生命周期内有效，无跨域风险
- 图片已在上传前经过 `compressImage()` 处理，尺寸受控

**范围外：**
- XSS 防护：OCR 文本以纯文本方式展示（使用 React 默认转义），不使用 `dangerouslySetInnerHTML`
- CSRF：不涉及新的 API 调用

### 配置

无新增配置项。所有行为通过组件 props 控制。

## 10. API 规范

**无新增后端 API。** 本需求完全复用现有接口：

- `POST /api/solve`：现有响应中的 `question` 字段即 OCR 识别文本
- 前端从 `SolveResponse.question` 读取 OCR 文本，无需后端任何修改

## 11. 成功标准

### MVP 成功定义

用户上传图片后能在聊天气泡中看到图片预览和 OCR 识别结果，消除"[图片]"黑盒感。

### 功能要求

- ✅ 图片消息气泡显示缩略图而非"[图片]"文字
- ✅ 点击缩略图弹出 Lightbox 查看大图
- ✅ 展开面板正确显示 OCR 识别文本
- ✅ 图片上传后缩略图立即显示（零延迟）
- ✅ OCR 加载中显示动画，完成后替换为文本

## 12. 实施阶段

### 阶段 1：基础数据流改造

**目标**：扩展 Message 类型，改造 use-chat Hook 以支持图片 URL 和 OCR 文本传递。

**交付物**：
- ✅ Message 接口新增 `imageUrl` 和 `ocrText` 字段
- ✅ use-chat 发送图片时生成并保存 blob URL
- ✅ use-chat 接收响应后回填 OCR 文本

**验证**：控制台 log 确认消息对象包含 imageUrl 和 ocrText 字段。

### 阶段 2：缩略图 + 展开面板

**目标**：实现图片缩略图展示和 OCR 结果展开面板。

**交付物**：
- ✅ ImageMessageBubble 组件（缩略图 + 展开/收起按钮 + OCR 文本面板）
- ✅ ChatMessage 集成 ImageMessageBubble
- ✅ 加载状态动画和错误提示

**验证**：上传图片后气泡显示缩略图，点击按钮展开 OCR 文本。

### 阶段 3：Lightbox 模态框

**目标**：实现图片放大查看功能。

**交付物**：
- ✅ ImageLightbox 组件（模态框 + 遮罩层 + 关闭交互）
- ✅ 键盘支持（ESC 关闭）
- ✅ 无图片时禁用点击

**验证**：点击缩略图弹出大图，支持三种关闭方式。

### 阶段 4：优雅降级与边缘情况

**目标**：处理异常场景。

**交付物**：
- ✅ 图片加载失败占位图
- ✅ blob URL 失效（页面刷新）时的优雅降级
- ✅ 纯文本消息（无图片）的兼容渲染

**验证**：刷新页面后图片区域显示占位图，纯文本消息渲染不受影响。

## 13. 未来考虑

- **图片持久化存储**：接入 Cloudflare R2，上传后返回永久 URL，解决页面刷新后图片丢失问题
- **OCR 文本可编辑**：允许用户修正识别错误并重新提交解题
- **多图支持**：支持一次上传多张图片（如试卷拍照分多张上传）
- **图片标注**：用户可在图片上圈选特定题目区域
- **历史会话图片回显**：结合 R2 持久化，在历史记录中完整回显图片

## 14. 风险与缓解措施

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| blob URL 页面刷新后失效 | 中 | 显示占位图 + 提示文字，OCR 文本仍可展示。后续版本通过 R2 持久化彻底解决 |
| 大图片导致内存占用高 | 低 | 已有 compressImage() 将图片压缩至 1536px，内存可控 |
| OCR 文本与实际题目不符导致用户困惑 | 中 | OCR 文本默认收起，用户主动展开查看；后续版本支持编辑修正 |
| Lightbox 在移动端体验不佳 | 低 | 使用 viewport 百分比单位（90vw/90vh），自适应屏幕尺寸 |
