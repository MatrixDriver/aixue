# AIXue MVP 测试规格与验收标准

基于 PRD（`rpiv/requirements/prd-aixue-mvp.md`）和测试策略（`rpiv/validation/test-strategy-aixue-mvp.md`）制定。

---

## 1. 单元测试规格

### 1.1 用户系统（US-08）

#### UT-USR-001: 用户注册 - 正常流程
- **前置条件**: 数据库为空
- **输入**: username="student01", password="Abc12345", name="张三", grade="初二", subjects=["数学", "物理"]
- **预期结果**: 用户创建成功，返回 User 对象，密码以 bcrypt 哈希存储
- **验收标准**: `user.id` 非空；`user.password_hash != "Abc12345"`；数据库中可查询到该用户

#### UT-USR-002: 用户注册 - 重复用户名拒绝
- **前置条件**: 已存在 username="student01" 的用户
- **输入**: username="student01", password="Xyz67890", name="李四", grade="初三"
- **预期结果**: 抛出 `DuplicateUserError`，数据库无新增记录
- **验收标准**: 异常类型和消息正确；数据库用户数量不变

#### UT-USR-003: 用户注册 - 必填字段校验
- **输入组合**:
  - 缺少 username → 抛出 `ValidationError`
  - 缺少 password → 抛出 `ValidationError`
  - 缺少 name → 抛出 `ValidationError`
  - 缺少 grade → 抛出 `ValidationError`
  - grade 非法值 "大一" → 抛出 `ValidationError`
- **验收标准**: 每种缺失字段均触发明确的校验错误；错误消息指明缺失字段名

#### UT-USR-004: 用户登录 - 正常流程
- **前置条件**: 已注册用户 student01
- **输入**: username="student01", password="Abc12345"
- **预期结果**: 返回 User 对象
- **验收标准**: 返回的用户信息与注册时一致

#### UT-USR-005: 用户登录 - 密码错误
- **输入**: username="student01", password="WrongPass"
- **预期结果**: 返回 None 或抛出 `AuthenticationError`
- **验收标准**: 不泄漏是用户名不存在还是密码错误的细节

#### UT-USR-006: 用户 Profile 更新
- **前置条件**: 已注册用户，年级="初二"
- **操作**: update_profile(grade="初三")
- **预期结果**: Profile 更新成功
- **验收标准**: 查询该用户的 grade 为 "初三"

#### UT-USR-007: 数据隔离 - 跨用户访问拒绝
- **前置条件**: 用户 A 和用户 B 各有解题记录
- **操作**: 以用户 A 身份查询用户 B 的解题历史
- **预期结果**: 返回空列表或抛出 `PermissionError`
- **验收标准**: 不返回用户 B 的任何数据

### 1.2 数据模型

#### UT-MDL-001: Problem 模型字段约束
- **测试**: 创建 Problem 对象
  - subject 仅接受 ["数学", "物理", "化学", "生物"]
  - difficulty 仅接受 1-5
  - knowledge_points 至少包含 1 项
- **验收标准**: 非法值触发 `ValidationError`

#### UT-MDL-002: Solution（SolveResult）模型结构
- **测试**: 创建 SolveResult 对象
  - steps 为 `list[SolveStep]`，每个 step 包含 step_number、content、knowledge_point
  - confidence 范围 0.0-1.0
  - verification_status 仅接受 ["verified", "unverified", "partial", "failed"]
- **验收标准**: 结构完整，字段类型正确

#### UT-MDL-003: ExamRecord 模型 - 对错标注
- **测试**: 创建含 10 道题的 ExamRecord，标注 7 对 3 错
- **验收标准**: `record.correct_count == 7`；`record.accuracy == 0.7`

#### UT-MDL-004: DiagnosisReport 模型 - 五维完整性
- **测试**: 创建 DiagnosisReport，检查五维分析字段
- **验收标准**: dimensions 包含恰好 5 个维度：知识漏洞、思维路径、概念关联、解题习惯、认知水平

#### UT-MDL-005: ConversationHistory 模型 - 消息序列化
- **测试**: 创建含 5 轮对话的 ConversationHistory
- **验收标准**: 消息按时间排序；每条消息含 role（user/assistant）和 content

### 1.3 SymPy 验证模块

#### UT-SYM-001: 一元一次方程求解
- **输入**: "2x + 3 = 7"
- **预期**: x = 2
- **验收标准**: SymPy 返回正确解，耗时 < 5 秒

#### UT-SYM-002: 一元二次方程求解
- **输入**: "x^2 - 5x + 6 = 0"
- **预期**: x = 2 或 x = 3
- **验收标准**: 返回两个解，均正确

#### UT-SYM-003: 二元一次方程组
- **输入**: "2x + y = 5, x - y = 1"
- **预期**: x = 2, y = 1
- **验收标准**: 两个变量的值均正确

#### UT-SYM-004: 不等式求解
- **输入**: "2x - 1 > 3"
- **预期**: x > 2
- **验收标准**: 返回正确的不等式解集

#### UT-SYM-005: 二次函数顶点计算
- **输入**: "y = x^2 - 4x + 3"
- **预期**: 顶点 (2, -1)
- **验收标准**: 计算正确

#### UT-SYM-006: 向量点积（垂直验证）
- **输入**: a = (1, 2), b = (-2, 1)
- **断言**: a·b == 0（垂直）
- **验收标准**: 断言通过

#### UT-SYM-007: 三角函数计算
- **输入**: "sin(pi/6)"
- **预期**: 1/2
- **验收标准**: 精确值匹配

#### UT-SYM-008: 超时处理
- **输入**: 构造一个 SymPy 难以在时限内求解的表达式
- **预期**: 在 SYMPY_TIMEOUT 秒内超时，返回降级标记
- **验收标准**: 不无限阻塞，返回超时状态

#### UT-SYM-009: 无法求解的降级
- **输入**: 几何证明类问题（SymPy 不适用）
- **预期**: 返回 `None` 或降级标记，不抛出异常
- **验收标准**: 降级标记明确，后续流程走 LLM-only 路径

### 1.4 Assert 断言验证

#### UT-AST-001: 断言构造 - 数值等式
- **输入**: 解题结果 x=2，标准答案 x=2
- **预期**: 断言通过
- **验收标准**: `assert_result.passed == True`

#### UT-AST-002: 断言构造 - 数值不等（验证失败）
- **输入**: 解题结果 x=3，标准答案 x=2
- **预期**: 断言失败
- **验收标准**: `assert_result.passed == False`；触发重试逻辑

#### UT-AST-003: 重试机制
- **模拟**: 断言连续失败 3 次
- **预期**: 重试 3 次后停止，返回最终失败结果并标注置信度降低
- **验收标准**: 重试次数 == MAX_RETRY_COUNT；最终状态为 "failed"

#### UT-AST-004: 重试成功
- **模拟**: 第 1 次断言失败，第 2 次成功
- **预期**: 返回第 2 次结果，状态为 "verified"
- **验收标准**: 最终结果来自第 2 次尝试

### 1.5 图片识别

#### UT-OCR-001: 图片格式校验
- **输入**: JPEG 图片 → 接受；PNG → 接受；WebP → 接受；GIF → 拒绝；PDF → 拒绝
- **验收标准**: 仅接受 PRD 中指定的格式

#### UT-OCR-002: 图片大小限制
- **输入**: 超大图片（>10MB）
- **预期**: 拒绝并提示文件过大
- **验收标准**: 返回明确的错误消息

#### UT-OCR-003: 模糊检测
- **输入**: 一张模糊的数学题目图片
- **预期**: 检测到模糊，提示重新上传
- **验收标准**: 返回 "image_quality_low" 状态

## 2. 集成测试规格

### 2.1 解题管线（数学深度模式）

#### IT-SOL-001: 完整解题管线 - 一元二次方程
- **输入**: 包含 "x^2 - 5x + 6 = 0" 的图片
- **流程**: 图片识别 → 学科判定（数学）→ SymPy 求解（x=2,3）→ LLM 生成步骤 → 断言验证
- **验收标准**:
  - 最终答案包含 x=2 和 x=3
  - verification_status == "verified"
  - steps 不为空，每步含 knowledge_point
  - 总耗时 < 30 秒

#### IT-SOL-002: 完整解题管线 - 几何题（SymPy 降级）
- **输入**: 一道三角形全等证明题
- **流程**: 图片识别 → 学科判定（数学）→ SymPy 返回 None → LLM-only 解题 → 标注置信度
- **验收标准**:
  - 解题步骤逻辑合理
  - verification_status == "unverified" 或 "partial"
  - confidence 标注为降低值

#### IT-SOL-003: 理化生解题管线
- **输入**: 一道力学分析题图片
- **流程**: 图片识别 → 学科判定（物理）→ LLM 直接解题 → 无验证
- **验收标准**:
  - 学科正确判定为物理
  - 解题步骤包含受力分析
  - 无 SymPy 调用

#### IT-SOL-004: 苏格拉底引导模式
- **输入**: 数学题 + mode="socratic"
- **验收标准**:
  - 第一轮输出不包含最终答案
  - 输出包含引导性提问（"你能...", "想一想...", "注意到..."等）
  - 逐步递进而非一次性揭示

#### IT-SOL-005: 完整解答模式
- **输入**: 同一数学题 + mode="full"
- **验收标准**:
  - 输出包含完整解题过程
  - 步骤编号连续
  - 每步标注知识点

#### IT-SOL-006: 多轮追问上下文保持
- **流程**:
  1. 第一轮：上传题目，获得解答
  2. 第二轮：追问 "为什么用这个公式？"
  3. 第三轮：追问 "有没有其他解法？"
- **验收标准**:
  - 每轮追问能感知前文上下文
  - 回答不重复已给信息
  - session_id 三轮一致

#### IT-SOL-007: 断言失败自动重试
- **模拟**: 构造一个 LLM 首次生成错误答案的场景（mock LLM 第一次返回错误值）
- **验收标准**:
  - 系统自动触发重试
  - 最终返回正确结果（第二次 mock 返回正确值）
  - 日志记录重试过程

### 2.2 年级个性化

#### IT-PER-001: 初二 vs 初三讲解深度
- **前置条件**: 两个测试用户，分别为初二和初三
- **输入**: 同一道数学题
- **验收标准**:
  - 初二用户的解答不引用初三以上知识点
  - 两份解答的语言复杂度存在差异（可由 LLM-as-Judge 评估）

#### IT-PER-002: Profile 注入验证
- **操作**: 检查发送给 LLM 的 Prompt
- **验收标准**: Prompt 中包含用户的年级和学科信息

### 2.3 数据库操作

#### IT-DB-001: 解题记录持久化
- **流程**: 用户解题 → 检查数据库
- **验收标准**: 数据库中保存了题目图片路径、识别文本、解题步骤、对话记录、验证状态

#### IT-DB-002: 学情数据积累
- **流程**: 同一用户解 10 道题（5 对 5 错），触发学情分析
- **验收标准**: 系统能获取到 10 条解题记录；正确率计算为 50%

#### IT-DB-003: 数据隔离集成验证
- **流程**: 用户 A 解 5 题，用户 B 解 3 题
- **验收标准**: 查询用户 A 历史仅返回 5 条；查询用户 B 历史仅返回 3 条

---

### 2.4 学情诊断集成

#### IT-DIA-001: 试卷导入完整流程
- **输入**: 一套含 10 道题的试卷图片
- **流程**: 图片识别 → 题目提取 → AI 解题 → 对错判定
- **验收标准**: 成功提取 >= 8 道题（识别率 >= 80%）；每道题有对错标记

#### IT-DIA-002: 五维诊断报告生成
- **前置条件**: 用户有 >= 20 条解题记录
- **验收标准**:
  - 报告包含五维分析，每个维度有具体内容
  - overall_score 在 0-100 范围内
  - weak_points 列表非空且按严重程度排序

#### IT-DIA-003: 练习题推荐
- **前置条件**: 诊断发现 "一元二次方程" 为薄弱点
- **验收标准**:
  - 推荐 5-10 道练习题
  - 每道题的 knowledge_points 包含 "一元二次方程"
  - 难度递进（基础 -> 巩固 -> 提升）
  - 不推荐用户已做过的题目

#### IT-DIA-004: 冷启动降级
- **前置条件**: 用户仅有 5 条解题记录（< 20）
- **验收标准**:
  - 使用 LLM 基础分析（非 pyKT）
  - 报告包含提示："数据量有限，诊断结果仅供参考"
  - 仍然给出初步的薄弱点分析

#### IT-LLM-001: API 超时处理
- **模拟**: LLM API 超时

#### IT-LLM-002: 流式输出完整性
- **操作**: 调用 LLM 解题接口，使用 SSE 流式返回

## 3. AI 输出质量测试规格

### 3.1 数学解题正确性

#### QA-MATH-001: 代数题测试集（10 题）
- **题型覆盖**: 一元方程、二元方程组、一元二次方程、不等式、因式分解
- **验收标准**: 最终答案正确率 >= 95%（即最多 0-1 题错误）

#### QA-MATH-002: 函数题测试集（8 题）
- **题型覆盖**: 一次函数、二次函数（图像/性质/应用）、反比例函数
- **验收标准**: 最终答案正确率 >= 95%

#### QA-MATH-003: 几何题测试集（7 题）
- **题型覆盖**: 三角形（全等/相似）、四边形、圆、坐标几何
- **验收标准**: 最终答案正确率 >= 90%（几何题 SymPy 覆盖有限，容许稍低）

#### QA-MATH-004: 统计与概率测试集（5 题）
- **题型覆盖**: 数据统计、概率计算
- **验收标准**: 最终答案正确率 >= 95%

#### QA-MATH-005: 高中数学测试集（20 题）
- **题型覆盖**: 集合、函数（对数/指数/三角）、向量、数列、立体几何、解析几何
- **验收标准**: 最终答案正确率 >= 90%

### 3.2 解题步骤质量

#### QA-STEP-001: 步骤逻辑性评估
- **方法**: 从测试集中选 10 题，对每题的解题步骤进行评分
- **评分维度**:
  - 步骤是否有逻辑跳跃（1-5 分）
  - 推理链是否完整（1-5 分）
  - 是否标注了使用的定理/知识点（是/否）
- **验收标准**: 平均得分 >= 4.0；知识点标注覆盖率 >= 80%

#### QA-STEP-002: LaTeX 语法正确性
- **方法**: 对所有测试集答案的 LaTeX 表达式进行语法检查
- **验收标准**: 无 LaTeX 语法错误（可被 KaTeX 正确渲染）

### 3.3 教学模式质量

#### QA-SOC-001: 苏格拉底引导质量
- **测试场景**: 5 道不同难度的数学题，使用苏格拉底模式
- **评估维度**:
  - 首轮是否直接给出答案（不应给出）
  - 提问是否有引导性
  - 提示是否逐步递进
  - 学生给出部分答案后是否给予肯定和下一步引导
- **验收标准**: LLM-as-Judge 评分 >= 3.5/5.0

#### QA-SOC-002: 追问处理质量
- **测试场景**: 3 个多轮追问对话（每个 3-5 轮）
- **评估维度**:
  - 是否准确理解追问意图
  - 回答是否与上下文一致
  - 是否重复已给信息
- **验收标准**: LLM-as-Judge 评分 >= 3.5/5.0

### 3.4 学情诊断质量

#### QA-DIA-001: 错因分类一致性
- **方法**: 5 套模拟试卷，每套预先标注错因类型
- **验收标准**: 系统错因分类与人工标注的一致率 >= 60%

#### QA-DIA-002: 弱项识别一致性
- **方法**: 5 套模拟试卷，每套预先标注 top-3 薄弱知识点
- **验收标准**: 系统识别的薄弱知识点与人工标注的交集 >= 2/3

#### QA-DIA-003: 推荐题目相关性
- **方法**: 基于诊断结果推荐的练习题，检查知识点覆盖
- **验收标准**: 推荐题目中 >= 70% 的题目覆盖了薄弱知识点

## 4. 端到端测试规格

### 4.1 核心流程

#### E2E-001: 新用户注册 → 首次解题
- **步骤**:
  1. 访问应用首页
  2. 点击注册，填写：用户名 "test_e2e_01"、密码、姓名 "E2E测试"、年级 "初二"、学科 ["数学"]
  3. 注册成功，自动跳转到主页
  4. 上传一道一元一次方程图片
  5. 等待解答生成
  6. 查看解答结果
- **验收标准**:
  - 注册过程无报错
  - 主页显示用户姓名和年级
  - 解答在 30 秒内返回
  - 解答包含解题步骤和答案
  - 解答中有 LaTeX 公式正确渲染

#### E2E-002: 拍照解题 - 苏格拉底模式
- **步骤**:
  1. 登录已有用户
  2. 上传数学题目图片
  3. 选择苏格拉底引导模式
  4. 查看 AI 引导提示
  5. 输入部分答案回复
  6. 查看 AI 进一步引导
  7. 继续对话直到获得完整解答
- **验收标准**:
  - AI 首轮不直接给出答案
  - AI 引导提示包含提问
  - 学生回复后 AI 给予反馈
  - 整个引导过程逻辑连贯

#### E2E-003: 拍照解题 - 完整解答模式
- **步骤**:
  1. 登录 → 上传数学题 → 选择完整解答模式
  2. 查看完整解题步骤
- **验收标准**:
  - 一次性返回完整解答
  - 步骤编号连续
  - 包含知识点标注

#### E2E-004: 模式切换
- **步骤**:
  1. 以苏格拉底模式开始解题
  2. 中途切换到完整解答模式
- **验收标准**: 切换后立即展示完整解题过程

#### E2E-005: 理化生解题
- **步骤**:
  1. 上传物理题图片
  2. 系统自动识别为物理
  3. 查看解答
- **验收标准**:
  - 学科判定正确
  - 解答包含物理概念和公式
  - 无 SymPy 验证标记

#### E2E-006: 试卷上传 → 学情诊断
- **步骤**:
  1. 进入学情诊断页面
  2. 上传一套试卷图片（多张）
  3. 系统自动识别题目和答案
  4. 用户确认/修正对错标注
  5. 查看生成的诊断报告
  6. 查看推荐练习题列表
  7. 点击一道推荐题查看详情
- **验收标准**:
  - 试卷上传支持多张图片
  - 识别出的题目可被用户确认
  - 诊断报告包含五维分析
  - 推荐练习题可点击查看

#### E2E-007: 解题历史查看
- **步骤**:
  1. 解完多道题后进入历史页面
  2. 查看历史列表
  3. 点击某条记录查看详情
- **验收标准**:
  - 历史按时间倒序排列
  - 每条记录显示题目摘要、学科、时间
  - 详情页包含完整对话记录和解题步骤

### 4.2 异常流程

#### E2E-ERR-001: 模糊图片处理
- **步骤**: 上传一张模糊不清的图片
- **验收标准**: 系统提示 "图片质量较低，请重新拍照上传"；不生成错误解答

#### E2E-ERR-002: 非题目图片处理
- **步骤**: 上传一张风景照片
- **验收标准**: 系统提示 "未识别到题目内容，请上传包含题目的图片"

#### E2E-ERR-003: 未登录访问
- **步骤**: 未登录状态下直接访问解题页面 URL
- **验收标准**: 重定向到登录页面

#### E2E-ERR-004: 网络中断恢复
- **步骤**: 解题过程中模拟网络中断（停止 LLM API mock server）
- **验收标准**: 前端显示友好的网络错误提示；提供重试按钮

#### E2E-ERR-005: 大文件上传拒绝
- **步骤**: 上传一个 15MB 的图片文件
- **验收标准**: 提示文件过大，建议压缩后重新上传
