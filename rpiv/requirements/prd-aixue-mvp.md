# AIXue MVP 产品需求文档

## 1. 执行摘要

AIXue（爱学）是一款 AI 驱动的个性化学习助手，面向初高中学生，解决日常学习中遇到难题缺少即时、高质量、零偏差讲解辅导的核心问题。产品以 Web 应用形态部署于 Railway 云端，兼顾手机拍照和电脑使用场景。

核心价值主张：通过多模态 LLM + 符号验证的双重保障，提供经过数学验证的解题辅导（数学学科），结合多维度学情诊断，实现"教会思考"而非"给答案"的教育目标。产品采用苏格拉底式引导与完整解答双模式，配合多轮追问，让学生在自主思考中掌握知识。

MVP 目标：为约 3 名初始用户（初二、初三学生）提供完整的智能解题 + 学情诊断 + 用户管理功能闭环，验证 AI 辅导教学的核心价值假设，不计成本追求解题质量。

## 2. 使命

### 产品使命

让每个学生都拥有一位耐心、精准、永不疲倦的 AI 学习伙伴，通过引导式教学培养独立思考能力，而非简单提供答案。

### 核心原则

1. **质量优先于速度**：解题结果必须经过验证，宁可慢一点也不能给出错误答案。AI 幻觉在教育场景中的误导成本极高
2. **引导优于灌输**：默认苏格拉底式引导，培养学生的思考能力，同时保留完整解答模式满足应试需求
3. **数据驱动的个性化**：基于学生的解题记录和错因分析，提供真正个性化的学习建议和练习推荐
4. **架构可演进**：初始用户量极小，但架构设计不做硬限制，为未来扩展留余地

## 3. 目标用户

### 主要用户画像

| 维度 | 描述 |
|------|------|
| **身份** | 初中和高中学生，当前重点初二和初三 |
| **人数** | MVP 阶段约 3 人（自己的孩子初二 + 两位同事的孩子初三） |
| **技术舒适度** | 熟悉手机拍照、浏览器操作，不需要复杂技术知识 |
| **使用场景** | 做作业遇到难题时拍照提问；考试后上传试卷分析弱项；日常复习查看学情报告 |
| **核心痛点** | 遇到难题无人即时辅导；不知道自己哪里薄弱；缺少针对性的练习题 |

### 关键需求

- 快速获得高质量、无错误的解题辅导
- 理解解题思路而非只看答案
- 了解自身知识薄弱点
- 获得针对性的练习推荐

## 4. MVP 范围

### 范围内

**核心功能**
- ✅ 智能解题：拍照/截图上传 -> AI 解析 -> 结构化解题步骤
- ✅ 双模式输出：苏格拉底式引导（默认）/ 完整解答（可切换）
- ✅ 多轮追问：基于对话上下文的连续交互
- ✅ 学科策略分层：数学深度模式（含验证）+ 理化生基础模式（纯 LLM）
- ✅ 数学三级交叉验证：SymPy 前置求解 + assert 断言 + 可选 Lean 4
- ✅ 学情诊断：多维错因分析 + 弱项报告 + 练习题推荐
- ✅ 试卷上传冷启动：快速积累初始学情数据
- ✅ 用户系统：多用户注册 + 年级 profile + 解题历史 + 数据隔离
- ✅ 个性化适配：根据年级 + 学科自动调整讲解深度

**题库**
- ✅ 基础题库：CMM-Math + TAL-SCQ5K-CN 数据集导入
- ✅ 日常积累：用户解题过程中自动入库
- ✅ LLM 生成变式题：基于错题生成针对性练习（CAS 验证正确性）

**技术**
- ✅ 多模态 LLM 端到端题目识别（主要方案）
- ✅ Pix2Text 作为 OCR 补充方案
- ✅ SymPy 数学验证引擎
- ✅ pyKT 知识追踪（simpleKT/AKT）
- ✅ LaTeX 公式渲染

**部署**
- ✅ Railway 云端部署
- ✅ 所有用户远程访问

### 范围外

**明确不做（第一版）**
- ❌ Manim 视频动画生成
- ❌ 语音交互（TTS / 语音输入）
- ❌ 学习路径规划
- ❌ 移动端原生 App
- ❌ 高并发优化
- ❌ 付费/商业化功能
- ❌ 家长端 / 教师端
- ❌ 理化生学科的符号验证和专项题库

## 5. 用户故事

### 核心用户故事

**US-01：拍照解题（苏格拉底模式）**
> 作为一名初二学生，我想要拍照上传一道数学难题并获得引导式提示，以便我能通过自己思考来理解解题方法。

示例：学生遇到一道二次方程应用题，拍照上传后，AI 先提问"你能从题目中找出哪些已知量？"，逐步引导学生列出方程并求解。

**US-02：拍照解题（完整解答模式）**
> 作为一名初三学生，我想要直接查看一道题的完整解题步骤，以便在时间紧迫时快速理解解法。

示例：考前复习时，学生切换到完整解答模式，上传一道几何证明题，直接获得包含定理标注的完整证明过程。

**US-03：多轮追问**
> 作为一名学生，我想要对 AI 的解答中不理解的步骤进行追问，以便彻底搞懂每一步的原因。

示例：AI 给出的解答中用到了"三角函数倍角公式"，学生追问"为什么这里要用倍角公式？"，AI 解释使用该公式的判断依据。

**US-04：理化生解题**
> 作为一名学生，我想要上传物理/化学/生物题目获得解答，以便获得全科辅导支持。

示例：学生上传一道力学分析题，AI 识别为物理学科，使用纯 LLM 推理给出受力分析和计算过程。

**US-05：试卷上传与学情冷启动**
> 作为一名新注册用户，我想要上传近期的考试试卷，以便系统快速了解我的知识水平。

示例：学生注册后上传期中数学试卷，系统自动识别题目和答案，判定对错，生成初始学情画像。

**US-06：学情诊断报告**
> 作为一名学生，我想要查看自己的学情诊断报告，以便了解哪些知识点需要加强。

示例：系统分析学生近期 30 道解题记录，生成五维诊断报告：知识漏洞（二次函数图像性质）、思维路径断裂（应用题建模环节）、概念关联缺失（函数与方程的联系）、解题习惯偏差（跳步计算）、认知水平（理解层→应用层过渡中）。

**US-07：针对性练习推荐**
> 作为一名学生，我想要获得针对我薄弱知识点的练习题推荐，以便进行有针对性的强化训练。

示例：诊断发现学生在"一元二次方程根与系数的关系"上薄弱，系统从题库中推荐 5 道相关练习题，难度从基础到进阶递进。

**US-08：用户注册与个性化**
> 作为一名学生，我想要注册账号并设置我的年级和学科信息，以便系统提供适合我水平的讲解。

示例：学生注册时填写"初二"、"重点学科：数学、物理"，之后所有解题和诊断都基于初二知识范围。

### 关键设计模式

1. **策略模式（学科解题）**：根据学科类型选择不同的解题策略（数学深度模式 vs 理化生基础模式），通过统一接口隔离实现差异
2. **管道模式（解题流程）**：输入识别 -> 学科判定 -> 策略选择 -> 解题生成 -> 验证 -> 输出格式化，各环节可独立替换
3. **会话管理模式**：每次解题创建会话上下文，支持多轮追问，会话内保持对话历史和状态
4. **Repository 模式（数据访问）**：统一的数据访问接口，底层可切换存储引擎

### 数据库选择

MVP 阶段使用 SQLite（文件型数据库），理由：
- 初始用户仅 3 人，无并发压力
- 部署简单，无需额外数据库服务
- Railway 持久存储支持 SQLite
- 后续可平滑迁移至 PostgreSQL

## 7. 功能规格

### 7.1 智能解题模块

#### 7.1.1 题目输入与识别

| 项目 | 规格 |
|------|------|
| 输入方式 | 拍照上传（手机相机）、截图上传、图片文件上传 |
| 图片格式 | JPEG、PNG、WebP |
| 识别方案 | 主：多模态 LLM 端到端识别；辅：Pix2Text OCR |
| 学科判定 | LLM 自动判定学科类型（数学/物理/化学/生物） |
| 异常处理 | 图片模糊时提示重新上传；无法识别时降级到手动输入 |

#### 7.1.2 数学解题（深度模式）

**三级交叉验证流程：**

```
题目输入
  |
  v
[SymPy 前置求解] -- 第一级
  | 获取数学真值
  v
[LLM 解题生成] <-- 真值作为硬约束注入 Prompt
  | 结构化解题步骤
  v
[assert 断言验证] -- 第二级
  | 关键步骤数值校验
  | 失败 -> 自动重试(最多3次)
  v
[Lean 4 验证(可选)] -- 第三级
  | 高难度题形式化证明
  v
输出解题结果
```

**输出格式：**
- 结构化分步解题（带步骤编号）
- LaTeX 公式渲染
- 每步标注使用的定理/知识点
- 置信度标注（AI 不确定时明确提示）

#### 7.1.3 理化生解题（基础模式）

- 使用多模态 LLM 直接推理
- 无额外符号验证
- 结构化分步输出
- 标注关键公式和概念

#### 7.1.4 对话模式

**苏格拉底式引导（默认）：**
- 不直接给出答案
- 通过提问引导学生思考
- 逐步揭示关键线索
- 学生回答后给予反馈和下一步引导
- Prompt 设计参考 Mr. Ranedeer 和 Khanmigo

**完整解答模式：**
- 直接给出完整解题过程
- 详细标注每一步的推理依据
- 适用于考前复习等时间紧迫场景

**多轮追问：**
- 保持会话上下文
- 支持对任意步骤的深入提问
- AI 能感知之前的对话内容给出连贯回答

#### 7.1.5 个性化适配

- 根据用户年级自动调整讲解深度
- 初二用户不引用高中知识点
- 将用户 profile 信息注入解题 Prompt

### 7.2 学情诊断模块

#### 7.2.1 数据采集

| 数据源 | 采集方式 | 用途 |
|--------|---------|------|
| 试卷上传 | 拍照上传，AI 自动识别题目和答案 | 冷启动，快速建立初始学情画像 |
| 系统解题记录 | 自动记录每次解题的题目、答案、耗时 | 持续积累学情数据 |
| 手动修正 | 学生可标注 AI 判定的对错是否准确 | 提高数据质量 |

#### 7.2.2 对错判定

- AI 自动解题并与学生答案比对
- 数学题使用 SymPy 辅助判定
- 支持学生手动修正判定结果

#### 7.2.3 五维错因分析

参考小猿五重错因分析框架：

| 维度 | 分析内容 | 技术实现 |
|------|---------|---------|
| 知识漏洞检测 | 缺失哪些知识点 | LLM 题目-知识点映射 + 错误模式统计 |
| 思维路径回溯 | 解题思路在哪一步断裂 | LLM 分析解题过程，定位断点 |
| 概念关联分析 | 相关联的知识点是否掌握 | 知识图谱遍历 + 关联知识点测试 |
| 解题习惯诊断 | 系统性的解题偏差 | 错误类型聚类分析（计算错误、概念混淆、跳步等） |
| 认知水平评估 | 当前认知层级（记忆→理解→应用→分析） | pyKT 知识追踪 + NeuralCD 认知诊断 |

#### 7.2.4 诊断报告

输出格式：
- 总体掌握度评分
- 各知识点掌握度雷达图
- 薄弱知识点排序列表
- 每个薄弱点的具体错因分析
- 针对性练习推荐（5-10 题/知识点）

#### 7.2.5 冷启动策略

- 数据不足阶段（< 20 道题）：LLM 做基础分析
- 数据充足后（>= 20 道题）：切换到 pyKT 模型进行知识追踪
- 过渡阶段：混合策略，逐步提高模型权重

#### 7.2.6 练习题推荐

- 基于薄弱知识点从题库匹配
- 难度递进（基础 -> 巩固 -> 提升）
- LLM 生成变式题（CAS 验证正确性）
- 已做过的题目不重复推荐

### 7.3 用户系统

#### 7.3.1 注册与登录

| 项目 | 规格 |
|------|------|
| 注册信息 | 用户名、密码、姓名、年级、重点学科 |
| 年级选项 | 初一至高三 |
| 学科选项 | 数学、物理、化学、生物（可多选） |
| 认证方式 | 用户名 + 密码（简单方案，MVP 够用） |

#### 7.3.2 用户 Profile

- 基本信息：姓名、年级、重点学科
- 解题统计：总题数、正确率、各学科分布
- 学情概览：当前薄弱知识点、认知水平
- Profile 信息自动注入到解题和诊断的 Prompt 中

#### 7.3.3 解题历史

- 完整保存每次解题记录（题目图片、识别文本、对话记录、结果）
- 按时间/学科/知识点筛选
- 支持回顾和重新学习

#### 7.3.4 数据隔离

- 每个用户的解题历史和学情数据完全隔离
- 用户只能查看自己的数据

### 7.4 题库系统

#### 7.4.1 基础题库

| 数据集 | 规模 | 覆盖范围 | 格式 |
|--------|------|---------|------|
| CMM-Math | 28K+ 题 | 小学至高中 | 多模态（含图片） |
| TAL-SCQ5K-CN | 5K 题 | 小/初/高 | LaTeX 格式 |

#### 7.4.2 题目数据模型

```python
class Problem:
    id: str
    subject: str            # 学科
    grade_level: str        # 适用年级
    knowledge_points: list  # 关联知识点
    difficulty: int         # 难度等级 1-5
    content: str            # 题目内容（LaTeX）
    solution: str           # 标准解答
    source: str             # 来源（数据集/用户解题/LLM生成）
    image_url: str | None   # 题目图片（如有）
```

#### 7.4.3 题库增长

- 用户解题过程中自动入库（去重）
- LLM 基于错题生成变式题
- 变式题经 CAS 验证正确性后入库

## 10. API 规格

### 内部 API 设计

MVP 以 Streamlit 为主体，API 以内部模块接口形式存在，不对外暴露 REST API。

#### 解题接口

```python
class SolverEngine:
    async def solve(
        self,
        image: bytes | None,      # 题目图片
        text: str | None,          # 题目文本（手动输入）
        subject: str | None,       # 学科（可选，自动判定）
        mode: Literal["socratic", "full"],  # 解答模式
        session_id: str | None,    # 会话ID（追问时传入）
        user_profile: UserProfile, # 用户信息
    ) -> SolveResult:
        ...

class SolveResult:
    steps: list[SolveStep]        # 解题步骤
    knowledge_points: list[str]   # 涉及知识点
    confidence: float             # 置信度
    verification_status: str      # 验证状态
    session_id: str               # 会话ID
```

#### 学情诊断接口

```python
class DiagnosisEngine:
    async def analyze(
        self,
        user_id: str,
        scope: Literal["full", "subject", "recent"],
        subject: str | None = None,
    ) -> DiagnosisReport:
        ...

    async def import_exam(
        self,
        user_id: str,
        images: list[bytes],       # 试卷图片
    ) -> ExamImportResult:
        ...

class DiagnosisReport:
    overall_score: float           # 总体掌握度
    dimensions: list[DimensionResult]  # 五维分析结果
    weak_points: list[WeakPoint]   # 薄弱知识点
    recommendations: list[Problem] # 推荐练习题
```

#### 用户服务接口

class UserService:

## 14. 风险与缓解措施

| 风险 | 影响 | 概率 | 缓解策略 |
|------|------|------|---------|
| **LLM 解题幻觉** | 给学生错误解答，损害信任 | 中 | 数学学科三级验证；理化生标注置信度；错误自动重试 |
| **OCR 识别不准** | 题目解析错误导致答非所问 | 中 | 多模态 LLM + Pix2Text 双重方案；识别结果可编辑修正 |
| **SymPy 无法处理的题型** | 几何证明、应用题等难以符号化 | 高 | SymPy 覆盖代数/计算类题型；几何/应用题降级为 LLM 多模型交叉验证 |
| **冷启动数据不足** | 学情诊断不准确 | 中 | 试卷上传快速积累；数据不足阶段用 LLM 基础分析过渡 |
| **LLM API 成本** | 不计成本但需关注 | 低 | 分级使用模型（诊断用 Haiku、解题用 Sonnet）；缓存相似题目的解答 |
