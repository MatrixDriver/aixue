# 图片预处理与 OCR/推理拆分成本优化 PRD

## 1. 执行摘要

AIXue 当前解题流程中，用户上传的高分辨率图片直接发送给视觉 LLM 进行 OCR 识别和解题推理，导致 token 消耗过高。同时，OCR 和推理共用同一模型，无法针对各阶段独立优化成本。

本需求通过三个维度降低单题 API 成本：前端图片自动压缩（减少图片 token）、后端 OCR/推理两阶段拆分（OCR 输出纯文本后推理阶段不再消耗图片 token）、模型独立配置（OCR 用轻量模型，推理用强力模型）。整个优化对终端用户完全透明，不影响使用体验。

**MVP 目标**：在不牺牲 OCR 识别质量和解题准确率的前提下，将单题图片解题的 API 成本降低 40%+。

## 2. 使命

**使命声明**：以最低成本提供高质量的 AI 解题服务，让教育 AI 的运营成本可持续。

**核心原则**：
1. **质量不妥协**：成本优化不能降低 OCR 识别准确率和解题质量
2. **用户无感知**：所有优化在后台完成，用户体验保持一致
3. **配置灵活**：通过环境变量独立配置各阶段模型，便于运营调优
4. **渐进式改造**：基于现有架构渐进修改，不做大规模重构

## 3. 目标用户

### 终端用户（初高中学生）
- **技术舒适度**：普通，使用手机拍照或截图上传题目
- **影响**：无感知，解题体验保持不变
- **关键需求**：快速获得准确的解题思路

### 运营人员
- **技术舒适度**：熟悉 .env 配置和服务部署
- **影响**：获得更灵活的模型配置能力
- **关键需求**：能独立调整 OCR 和推理模型以优化成本

## 4. MVP 范围

### 范围内

**前端图片处理：**
- ✅ 上传图片自动缩放至最长边 1536px
- ✅ 自动转换为 WebP 格式（quality=85）
- ✅ 浏览器不支持 WebP 时降级为 JPEG（quality=80）
- ✅ 小于 1536px 的图片不放大，仅做格式转换

**后端 OCR/推理拆分：**
- ✅ OCR 阶段：接收图片，使用视觉 LLM 输出题目结构化文本（含 LaTeX 公式）
- ✅ 推理阶段：接收纯文本，使用推理 LLM 输出解题思路和答案
- ✅ 纯文本输入自动跳过 OCR 阶段

**配置管理：**
- ✅ .env 新增 `LLM_MODEL_OCR` 配置项
- ✅ config.py Settings 类新增对应字段
- ✅ `LLM_MODEL_OCR` 未配置时回退到 `LLM_MODEL_LIGHT`

### 范围外

- ❌ 手动裁剪功能
- ❌ 传统 OCR（PaddleOCR/Tesseract）方案
- ❌ 按学科/难度分模型
- ❌ 管理后台动态切换模型
- ❌ 成本数据持久化到数据库
- ❌ OCR 识别质量自动检测与降级/升级策略
- ❌ 成本统计面板

## 5. 用户故事

### US-1: 高清图片自动压缩
**作为**一个用手机拍照上传题目的学生，**我希望**系统自动处理图片大小，**以便**我不需要关心图片是否太大，同时获得快速响应。

**示例**：学生用 4000x3000 的手机拍照上传数学题，前端自动压缩至 1536x1152 的 WebP 格式后发送，减少约 60% 的数据传输量。

### US-2: 截图格式优化
**作为**一个通过截图上传题目的学生，**我希望**截图被自动优化，**以便**即使截图分辨率较低也能正常识别。

**示例**：学生截图 800x600 的题目图片，前端不放大图片，仅转换为 WebP 格式以减少体积。

### US-3: OCR/推理拆分透明化
**作为**一个上传图片解题的学生，**我希望**系统能准确识别题目并给出解答，**以便**我不需要关心后台是一步还是两步完成。

**示例**：学生上传含复杂数学公式的图片，系统先用 OCR 模型提取 LaTeX 文本，再用推理模型生成完整解题过程，最终展示与之前一致的解答格式。

### US-4: 纯文本直接解题
**作为**一个直接输入文字题目的学生，**我希望**系统直接解答，**以便**不浪费时间在不必要的 OCR 环节。

**示例**：学生输入"求 x^2 + 3x - 4 = 0 的解"，系统跳过 OCR 阶段直接进入推理解题。

### US-5: 模型独立配置
**作为**运营人员，**我希望**能分别配置 OCR 和推理使用的模型，**以便**用最经济的组合达到最佳性价比。

**示例**：运营在 .env 中设置 `LLM_MODEL_OCR=google/gemini-2.0-flash-001`（便宜的视觉模型做 OCR），`LLM_MODEL=google/gemini-2.5-pro-preview-03-25`（强力模型做推理）。

### US-6: WebP 降级兼容
**作为**使用旧浏览器的学生，**我希望**图片上传功能正常工作，**以便**我不会因为浏览器兼容性问题无法使用。

**示例**：某些旧版浏览器不支持 WebP 导出，前端自动降级为 JPEG 格式（quality=80）上传。

## 6. 核心架构与模式

### 高级架构

```
用户上传图片
    ↓
[前端] Canvas API 压缩 (1536px + WebP/JPEG)
    ↓
[后端 API] /api/solver/solve
    ↓
[SolverService] 判断输入类型
    ├── 图片输入 → OCR 阶段 (LLM_MODEL_OCR) → 结构化文本
    │                                              ↓
    └── 纯文本输入 ──────────────────────────→ 推理阶段 (LLM_MODEL)
                                                   ↓
                                              解题结果
```

### 关键设计模式

1. **阶段分离模式**：OCR 和推理作为独立阶段，通过结构化文本（含 LaTeX）作为中间数据传递
2. **配置回退模式**：`LLM_MODEL_OCR` 未配置时回退到 `LLM_MODEL_LIGHT`，保证向后兼容
3. **前端透明压缩**：压缩逻辑封装在图片上传工具函数中，不影响上传组件的 API

### 目录结构影响

```
src/aixue/
  config.py              — 新增 LLM_MODEL_OCR 字段
  services/
    ocr_service.py       — 修改：使用独立的 OCR 模型配置
    solver_service.py    — 修改：OCR 输出纯文本后传给推理阶段
    math_solver.py       — 修改：接收纯文本而非图片
    general_solver.py    — 修改：接收纯文本而非图片
    llm_service.py       — 可能需要适配：支持按调用指定模型

frontend/src/
  [图片上传组件]          — 新增压缩逻辑
  utils/imageCompress.ts — 新建：图片压缩工具函数
```

## 7. 功能规格

### F-1: 前端图片压缩

**目的**：在发送前压缩用户上传的图片，减少传输量和后端图片 token 消耗。

**操作流程**：
1. 用户选择图片文件
2. 创建 Image 对象加载原图
3. 计算缩放比例：`scale = min(1, 1536 / max(width, height))`
4. 创建 Canvas 绘制缩放后的图片
5. 尝试导出 WebP（quality=0.85），不支持则降级 JPEG（quality=0.80）
6. 将结果 Blob 作为上传数据

**关键约束**：
- 不放大小图片（scale 最大为 1）
- WebP 支持检测：检查 `canvas.toDataURL('image/webp')` 是否返回 `data:image/webp` 前缀
- 压缩为异步操作，使用 Promise 封装

### F-2: OCR 阶段独立化

**目的**：将 OCR 识别作为独立阶段，使用专门配置的模型。

**操作流程**：
1. `SolverService` 判断输入包含图片
2. 调用 `OCRService` 并指定使用 `LLM_MODEL_OCR` 模型
3. OCR 输出结构化文本（含 LaTeX 公式标记）
4. 将 OCR 文本结果传递给推理阶段

**关键约束**：
- OCR 结果为空时，返回错误提示"请上传更清晰的图片"
- OCR 阶段不做解题，只做文字和公式识别
- 保留现有的"聚焦指定题目"功能

### F-3: 推理阶段纯文本化

**目的**：推理阶段仅接收纯文本，不再消耗图片 token。

**操作流程**：
1. 接收 OCR 阶段输出的结构化文本（或用户直接输入的纯文本）
2. 调用推理模型（`LLM_MODEL`）进行解题
3. 数学题走 MathSolver（SymPy 验证流程不变），其他走 GeneralSolver

**关键约束**：
- 推理 Prompt 需要适配，明确告知 LLM 输入来自 OCR 识别结果
- SymPy 前置求解流程不变，仍基于 OCR 提取的 LaTeX 公式

### F-4: 模型配置独立化

**目的**：支持 OCR 和推理使用不同模型。

**配置项**：
| 环境变量 | 用途 | 默认回退 |
|---|---|---|
| `LLM_MODEL_OCR` | OCR 阶段模型 | 回退到 `LLM_MODEL_LIGHT` |
| `LLM_MODEL` | 推理阶段模型 | 无回退（必填） |

## 8. 技术栈

基于现有项目技术栈，无新增依赖：

- **后端**：FastAPI + Python（现有）
- **前端**：Next.js 16 + React 19 + TypeScript（现有）

- **配置管理**：pydantic-settings + .env（现有）

## 9. 安全与配置

### 配置管理

**新增环境变量**：
```bash
# .env
LLM_MODEL_OCR=google/gemini-2.0-flash-001  # OCR 专用模型（可选，回退到 LLM_MODEL_LIGHT）
```

**config.py 修改**：
```python
class Settings(BaseSettings):
    # 现有字段
    LLM_MODEL: str
    LLM_MODEL_LIGHT: str
    # 新增
    LLM_MODEL_OCR: str | None = None  # None 时回退到 LLM_MODEL_LIGHT

    @property
    def ocr_model(self) -> str:
        return self.LLM_MODEL_OCR or self.LLM_MODEL_LIGHT
```

### 安全考虑
- 前端图片压缩在客户端完成，不涉及服务端安全
- 模型配置通过 .env 管理，不暴露给前端
- OCR 中间文本仅在服务端内存传递，不持久化

### 部署
- 部署平台：Railway（Docker），与现有一致
- 无需新增基础设施
- 仅需在 Railway 环境变量中添加 `LLM_MODEL_OCR`（可选）

## 10. API 规范

本优化不新增 API 端点，现有 `/api/solver/solve` 接口保持不变。

**请求格式**：不变
```json
{
  "image": "base64_encoded_image",  // 前端已压缩
  "text": "题目文本",               // 可选
  "question_number": 1,             // 可选，聚焦题号
  "mode": "direct"                  // socratic | direct
}
```

**响应格式**：不变 — 对前端完全透明。

**内部变化**：
- 图片输入时，后端分两步调用 LLM（OCR + 推理），而非原来的一步
- 纯文本输入时，跳过 OCR 直接推理（与现有行为一致）

## 11. 成功标准

### MVP 成功定义
在保持解题质量不变的前提下，显著降低单题 API 成本。

### 功能要求
- ✅ 前端图片压缩功能正常工作（1536px 缩放 + WebP/JPEG 转换）
- ✅ OCR 阶段独立运行，正确提取题目文本和 LaTeX 公式
- ✅ 推理阶段仅接收纯文本，不再发送图片给推理模型
- ✅ 纯文本输入跳过 OCR 阶段
- ✅ `LLM_MODEL_OCR` 配置生效，未配置时正确回退
- ✅ 数学题 SymPy 验证流程不受影响

### 质量指标
- OCR 识别准确率：与优化前一致（人工抽检 20 题对比）
- 解题正确率：与优化前一致（人工抽检 20 题对比）
- 单题 API 成本：降低 40%+（通过日志中的 token 消耗对比）

### 用户体验目标
- 用户无感知：解题流程、响应格式、响应时间无明显变化
- 响应时间：由于增加 OCR 阶段，总时间可能略增（可接受范围内，不超过原来的 1.5 倍）

## 13. 未来考虑

- **管理后台动态模型切换**：运营可在 Web 界面实时切换 OCR/推理模型，无需重启服务
- **OCR 质量自动检测**：识别 OCR 结果质量，低于阈值时自动升级到更强模型重试
- **成本统计面板**：持久化 token 消耗数据，提供可视化成本分析
- **按学科/难度智能选模型**：数学用强力推理模型，语文/英语用轻量模型
- **多图拼接优化**：多张图片合并为一次 OCR 调用

## 14. 风险与缓解措施

| 风险 | 影响 | 缓解策略 |
|---|---|---|
| OCR 阶段信息丢失（如图形、表格） | 推理阶段缺少关键视觉信息，解题质量下降 | OCR Prompt 要求输出图形/表格的文字描述；后续版本可考虑 OCR 质量检测 |
| 两阶段调用增加总延迟 | 用户等待时间变长 | OCR 使用轻量快速模型；监控延迟数据，设定 1.5 倍上限 |
| WebP 兼容性问题 | 少数旧浏览器图片处理异常 | 实现 JPEG 降级方案；Canvas API 兼容性覆盖率 >98% |
| 图片压缩导致文字模糊 | OCR 识别率下降 | 1536px 阈值经过测试，足以保证文字清晰；quality=85 保持高画质 |
