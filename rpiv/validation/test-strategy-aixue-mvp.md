# AIXue MVP 测试策略

## 1. 测试目标与原则

### 1.1 测试目标
- 确保三大核心场景（智能解题、学情诊断、用户系统）功能正确且稳定
- 验证数学解题的三级交叉验证管线输出正确性
- 保障多用户数据隔离和安全性
- 验证 AI 输出质量满足教学标准

### 1.2 测试原则
- **质量优先**：初始用户仅 3 人，不考虑压力测试，聚焦功能正确性和 AI 输出质量
- **分层验证**：单元测试 → 集成测试 → AI 质量验证 → 端到端测试，逐层推进
- **可复现性**：AI 相关测试使用固定 seed / 缓存响应，确保结果可复现
- **最小覆盖**：MVP 阶段聚焦核心路径，不追求覆盖率指标

## 2. 单元测试策略

### 2.1 API 端点测试

| 模块 | 测试范围 | 关键测试点 |
|------|----------|------------|
| 用户注册/登录 | POST /register, POST /login | 参数校验（姓名、年级必填）、重复注册拒绝、JWT token 生成与验证 |
| 用户 Profile | GET/PUT /profile | 年级更新、学科偏好保存、未授权访问拒绝 |
| 题目上传 | POST /solve | 图片格式校验（jpg/png/webp）、文件大小限制、学科参数验证 |
| 解题结果 | GET /solve/{id} | 结果状态查询、不存在的 ID 返回 404、跨用户访问拒绝 |
| 追问对话 | POST /solve/{id}/followup | 上下文关联验证、空消息拒绝 |
| 试卷上传 | POST /exam | 多图上传、对错标注接口 |
| 学情报告 | GET /diagnosis/{user_id} | 数据不足时的降级响应、报告结构完整性 |
| 题库查询 | GET /problems | 按知识点/难度/年级筛选、分页参数验证 |

**技术选型**：pytest + httpx（异步 API 测试）

### 2.2 数据模型测试

| 模型 | 测试点 |
|------|--------|
| User | 必填字段验证、年级枚举值约束、密码哈希存储（不存储明文） |
| Problem | 题目元数据完整性、学科/知识点/难度字段约束 |
| Solution | 解题步骤结构化存储、LaTeX 格式保留、验证状态字段 |
| ExamRecord | 试卷-题目关联、对错标注、分数计算 |
| DiagnosisReport | 多维度诊断数据结构、知识点覆盖率计算 |
| ConversationHistory | 多轮对话上下文序列化、消息角色标注 |

### 2.3 验证管线单元测试（数学学科）

| 验证级别 | 测试点 |
|----------|--------|
| L1: SymPy 前置求解 | 方程求解正确性、表达式简化、不等式求解、向量运算、几何计算 |
| L2: Assert 断言 | 断言表达式构造正确性、断言失败时的重试触发逻辑 |
| L3: Lean 4（可选） | Lean 代码生成语法正确性、编译通过率 |
| 重试机制 | 最多 3 次重试、重试间隔、重试后结果替换逻辑 |

**关键测试用例**：
- SymPy 对多种题型的求解覆盖：一元方程、二元方程组、不等式、二次函数、三角函数、几何证明辅助计算
- 断言构造：向量点积=0 验证垂直、距离公式验证等
- 边界情况：SymPy 无法求解时的降级处理、超时处理

### 2.4 图片识别单元测试

| 测试点 | 说明 |
|--------|------|
| 多模态 LLM OCR | 数学公式图片识别准确性、手写体识别、印刷体识别 |
| Pix2Text 补充识别 | 中文数学混排识别、LaTeX 输出格式正确性 |
| 图片预处理 | 旋转校正、模糊检测（低质量图片提示重新上传） |

## 3. 集成测试策略

### 3.1 LLM API 调用测试

| 测试场景 | 验证点 |
|----------|--------|
| Prompt 注入 | SymPy 求解结果正确注入到 LLM Prompt 中 |
| 年级适配 | 不同年级 profile 注入后，LLM 输出的讲解深度差异 |
| 模式切换 | 苏格拉底引导模式 vs 完整解答模式的 Prompt 差异及输出格式 |
| 多轮对话 | 上下文正确传递、历史消息截断策略 |
| 异常处理 | API 超时重试、rate limit 处理、模型不可用降级 |
| 流式输出 | SSE 流式响应的完整性、中断恢复 |

**Mock 策略**：
- 使用 LLM 响应缓存（基于 Prompt hash）确保可复现性
- 关键路径使用真实 API 调用（标记为 `@pytest.mark.integration`）
- 非关键路径使用预录制的 mock 响应

### 3.2 数据库操作测试

| 测试场景 | 验证点 |
|----------|--------|
| 用户 CRUD | 创建、读取、更新、删除用户数据的完整流程 |
| 解题记录持久化 | 解题过程和结果正确存储、关联用户 ID |
| 学情数据积累 | 多次解题后诊断数据正确聚合 |
| 数据隔离 | 用户 A 无法访问用户 B 的解题记录和诊断报告 |
| 事务一致性 | 解题 + 记录 + 知识点标注在同一事务中 |

**测试数据库**：使用 SQLite 内存数据库或独立的测试数据库实例，每个测试用例前重置

### 3.3 验证管线集成测试

| 测试场景 | 验证点 |
|----------|--------|
| 完整管线 | 题目输入 → SymPy 求解 → Prompt 注入 → LLM 生成 → 断言验证 → 最终输出 |
| 失败重试 | 断言失败 → 自动重试 → 重新生成 → 再次验证 |
| 降级路径 | SymPy 无法求解 → 仅依赖 LLM → 标注置信度降低 |
| 理化生路径 | 非数学学科直接走 LLM 路径，无符号验证 |

## 4. AI 输出质量验证策略

### 4.1 解题正确性验证

**方法论**：构建标注测试集（Ground Truth），对比 AI 输出与正确答案

| 维度 | 评估方法 | 通过标准 |
|------|----------|----------|
| 最终答案正确性 | 与标准答案精确比对（数值/表达式等价判定） | 数学 >= 95%，理化生 >= 85% |
| 解题步骤逻辑性 | 人工审查 + LLM-as-Judge 辅助评分 | 步骤无逻辑跳跃，推理链完整 |
| 知识点标注准确性 | 与标注的知识点对比 | 召回率 >= 80% |
| LaTeX 渲染正确性 | 自动化检查 LaTeX 语法 + 渲染验证 | 无语法错误 |

**标注测试集构建**：
- 初中数学：30 题（覆盖代数、几何、函数、统计四大板块）
- 初中理化生：各 10 题
- 高中数学：20 题（覆盖主要知识模块）
- 题目来源：CMM-Math / TAL-SCQ5K-CN 中选取 + 人工标注解题过程

### 4.2 教学模式验证

| 模式 | 验证点 |
|------|--------|
| 苏格拉底引导 | 不直接给出答案、通过提问引导思考、提示逐步递进 |
| 完整解答 | 步骤完整、公式正确、解释清晰 |
| 追问处理 | 准确理解追问意图、回答与上下文一致 |

**评估方式**：
- 构建 10 个典型追问对话场景
- LLM-as-Judge 评估引导质量（1-5 分制）
- 人工抽样审查

### 4.3 学情诊断质量验证

| 维度 | 验证点 |
|------|--------|
| 错因分类准确性 | 五维错因分析的分类是否合理 |
| 弱项识别准确性 | 与人工标注的弱项对比 |
| 练习题推荐相关性 | 推荐题与弱项知识点的匹配度 |

**评估方式**：
- 构建 5 套模拟试卷（含已标注的错因和弱项）
- 对比系统诊断结果与人工标注的一致性
- 推荐题目的知识点覆盖率 >= 70%

### 4.4 年级适配验证

| 验证点 | 方法 |
|--------|------|
| 初二 vs 初三讲解深度 | 同一题目不同年级用户请求解答，对比讲解用词和深度 |
| 知识点引用范围 | 初二用户不应引用初三及以上知识点 |

## 5. 端到端测试策略

### 5.1 核心用户流程

| 流程 | 步骤 | 验证点 |
|------|------|--------|
| 注册 → 首次解题 | 注册填写年级 → 上传题目图片 → 等待解答 → 查看结果 | 全流程无报错，结果包含完整解题步骤 |
| 拍照解题（苏格拉底） | 上传图片 → 选择苏格拉底模式 → 查看引导 → 追问 → 获得更多提示 | 引导逻辑递进，追问上下文连贯 |
| 拍照解题（完整解答） | 上传图片 → 选择完整解答 → 查看解题过程 | 步骤完整，答案正确 |
| 模式切换 | 苏格拉底模式中途切换到完整解答 | 切换后展示完整解题过程 |
| 试卷上传 → 诊断 | 上传试卷图片 → 标注对错 → 查看诊断报告 → 查看推荐练习 | 报告包含五维分析，推荐题目可查看 |
| 多轮追问 | 解题后连续追问 3 轮 | 每轮追问上下文连贯，不出现"遗忘"前文 |
| 历史记录查看 | 解完题后 → 查看解题历史列表 → 点击查看详情 | 历史按时间排序，详情完整保留 |

### 5.2 异常流程

| 流程 | 步骤 | 验证点 |
|------|------|--------|
| 模糊图片 | 上传低质量图片 | 系统提示重新上传，不产生错误解答 |
| 非题目图片 | 上传风景照 | 系统识别非题目内容，友好提示 |
| 验证失败 | 数学题 SymPy 验证失败（构造会失败的场景） | 重试机制正常工作，最终给出标注置信度的结果 |
| 网络中断 | 解题过程中模拟网络中断 | 前端友好提示，可重试 |
| 未登录访问 | 未登录状态访问解题页面 | 重定向到登录页 |

### 5.3 测试工具
- **Web UI 测试**：Playwright（Python 版）
- **API 测试**：httpx + pytest
- **截图对比**：用于 UI 回归测试（可选，MVP 阶段优先级低）

## 6. 测试环境与数据

### 6.1 测试环境
- **单元测试 / 集成测试**：本地开发环境，SQLite 内存数据库
- **端到端测试**：本地启动完整服务栈（或 Docker Compose）
- **AI 质量测试**：需要真实 LLM API 调用，单独标记，可选执行

### 6.2 测试数据
- **用户数据**：预置 3 个测试用户（初二、初三各一个 + 管理员）
- **题目数据**：标注测试集（见 4.1），按学科和难度分类
- **试卷数据**：5 套模拟试卷图片 + 标注答案

### 6.3 CI/CD 集成
- 单元测试和集成测试（mock 模式）在每次 push 时运行
- AI 质量测试在 PR 合并前手动触发
- 端到端测试在部署前运行

## 7. 测试优先级

| 优先级 | 测试类别 | 理由 |
|--------|----------|------|
| P0 | 数学验证管线单元测试 | 核心差异化功能，正确性直接影响产品可信度 |
| P0 | 解题正确性验证 | 产品核心价值 |
| P0 | 用户数据隔离测试 | 多用户安全基础 |
| P1 | API 端点单元测试 | 接口契约保障 |
| P1 | LLM 集成测试 | 确保 Prompt 工程有效性 |
| P1 | 核心 E2E 流程 | 用户体验保障 |
| P2 | 学情诊断质量验证 | 功能重要但容忍度较高 |
| P2 | 教学模式验证 | 可通过人工审查补充 |
| P2 | 异常流程 E2E | 边缘场景，初期可接受人工处理 |

## 8. 风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| LLM 输出不确定性导致测试不稳定 | AI 质量测试通过率波动 | 使用响应缓存 + 统计通过率（多次运行取均值） |
| SymPy 无法覆盖所有题型 | 部分数学题缺少 L1 验证 | 记录 SymPy 覆盖率，不可解时走降级路径并标注 |
| 标注测试集规模有限 | 评估结果可能不具代表性 | 持续扩充测试集，优先覆盖用户实际遇到的题型 |
| 图片识别质量受拍照条件影响 | OCR 错误导致解题错误 | 测试集包含多种拍照质量（清晰、模糊、倾斜、阴影） |
